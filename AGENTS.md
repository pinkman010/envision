# Envision 项目约束

## 1. 项目目标与 P0 边界

- P0 交付“单报告 ESG 披露分析 MVP”：以远景能源 2024 中文 ESG 报告为输入，输出条款级证据、披露判断、差距建议、人工复核结果和审计记录。
- P0 标准采用按版本和生效日期管理的 GRI profile：报告期适用条款用于当前披露分析；截至 2026 年已生效的新条款用于准备度分析；2027 年及以后生效的条款仅作观察项。
- P0 操作界面为本仓库的 Streamlit。不得将 `C:\Users\43480\Desktop\esg-dashboard` 的静态数据或页面表述为 P0 已打通能力。
- `multi_company_benchmark`、`batch_corpus_processing`、实质性议题跨公司对标和 Claw 舆情监测均不属于当前 P0 已实现能力。

## 2. 事实边界

- 当前真实主流程仅为 `single_report_analysis`：`CorpusAgent -> RetrievalAgent -> AnalystAgent -> AdvisorAgent`。
- AI 输出只能作为分析辅助。没有报告证据时，结论必须进入 `manualReview`，不得补造企业披露事实。
- 每条披露判断必须可追溯至条款 ID、条款版本、证据原文、报告页码、运行 ID 和人工复核状态。
- 不得将静态演示数据、规划功能或未经人工标注验证的指标写成真实结果。

## 3. 数据与来源

- P0 报告基线：`data/knowledge_base/peer_reports/Envision Energy 2024-zh.pdf`。
- GRI 原始资料：`data/knowledge_base/standards/gri_reference/GRI_Standards_Official_Consolidated_Set_en.pdf`。该合订本含 2021 通用标准和后续修订条款，只能作为原始参考资料；实际 P0 范围以后续 `p0_gri_manifest` 为准。
- 原始报告和标准文件视为证据材料。不得覆盖、删改或用处理后文本替代原始文件。
- 导入、分块和检索数据至少保留 `company`、`year`、`industry`、`topic`、页码、分块 ID 和来源文件哈希。

## 4. 依赖与运行环境

- `pyproject.toml` 是 Python 依赖的唯一声明来源；`environment.yml` 通过 `pip install -e ".[dev]"` 安装该项目。不得为同一依赖维护第二份手工清单。
- 项目验证使用 `C:\ProgramData\miniconda3\envs\envision\python.exe`。Codex Bundled Python 不代表项目运行环境，不能据此判断项目依赖缺失。
- `python-dotenv` 已声明，用于读取 `.env`；不得重复添加。
- `PyPDF2` 仍服务于当前主语料解析器；`pypdf` 服务于 `scripts/import_peer_reports.py`。两者在完成解析器统一前均应保留。
- 当前环境的 `pip check` 存在 `numpy/scipy/torch` 等非项目直接依赖冲突。除非复现到项目功能，不要通过在本项目新增无关依赖或全局降级来掩盖该问题。
- 不提交 `.env`、密钥、运行日志中的敏感配置或外部服务响应中的非公开数据。

## 5. 主代理与子代理

- 主代理只负责项目规划、范围控制、架构决策、证据审查、阶段验收和结果整合，避免在长期执行细节中消耗主上下文。
- 代码、数据整理、测试、文档核验等边界清晰且可独立验证的执行任务，优先交由子代理完成。
- 子代理必须先读取本文件和相关局部约束，只在授权范围内修改，并回传修改文件、验证结果、风险与未完成项。
- 主代理负责复核子代理结果，确认事实边界、依赖影响和验收条件后，才能进入下一阶段。
- 不能把阻塞当前决策的关键任务完全委派给子代理；主代理必须保留技术判断与最终决策权。
- 派发子代理时默认使用最小上下文；除非任务明确需要完整历史，否则使用 `fork_turns="none"` 或只提供必要片段，避免子代理误判自己是主代理或调度者。
- 子代理执行任务时不得使用 `list_agents`、`wait_agent`、`send_message` 等协作调度工具；这些工具只由主代理用于协调。
- 子代理必须按节点回报：已读取上下文、已完成写入、已完成验证。主代理在中断子代理前，必须先检查目标产物是否已经落地，避免覆盖已完成结果。
- 涉及中文路径、长 JSON、长 Python 逻辑或复杂 PowerShell 引号传递时，优先使用 UTF-8 无 BOM 临时脚本执行，并在完成后清理临时文件；避免复杂内联命令导致路径编码或引号错误。
- 子代理超时后，主代理必须区分“未开始写入”“写入完成但未回报”“权限/编码/路径失败”三类情况，再决定是否重派、接管或回滚。

## 6. 实施顺序

1. 按 `docs/ESG项目全生命周期实施计划.md` 完成阶段 A：来源核验与 GRI profile 冻结。
2. 定义并持久化 `AnalysisRun`、`DisclosureAssessment`、`Evidence`、`ReviewDecision`。
3. 建设双语 GRI 条款库与可追溯报告证据层。
4. 再修改 Agent、Prompt、规则和 Streamlit 页面。
5. 完成真实运行、人工标注评测和误差复盘后，更新结项材料。

未通过当前阶段门时，不进入下一阶段；不得为视觉展示提前开发 P1/P2 模块。

## 7. 修改与验证

- 每次修改前先说明影响的文件、数据、运行行为和验证方式；涉及外部模型调用、知识库重建、文件导入或删除时，先取得用户确认。
- 优先做最小改动。保留用户已有的未提交改动，不回退无关文件。
- 当前仓库没有 `tests/` 目录。新增逻辑时补充最小可重复测试；现有逻辑变更至少验证相关模块导入、数据契约和页面/API 行为。
- 对 PDF 解析需同时检查文本提取、页码对应和 GRI 索引表格结构。纯文本提取会丢失部分表格单元格边界，不能直接作为条款清单的唯一来源。

## 8. 文档同步

- 项目计划文件默认使用中文撰写；代码标识符、命令、路径、JSON 字段名和枚举值保持英文，避免破坏可执行性。
- 生命周期范围、阶段门和关键决策更新到 `docs/ESG项目全生命周期实施计划.md`。
- 已实现架构更新到 `docs/项目模型说明文档.md`；用户可操作步骤更新到 `docs/用户操作手册.md`；运行记录和验证结果更新到 `docs/DEV_LOG.md`。
- 规划能力必须标注为“规划中”或“未实现”，不得写入已实现清单。
