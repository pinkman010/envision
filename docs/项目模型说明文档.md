# 项目模型说明文档

## 1. 文档目的

本文档描述当前仓库中的实际系统模型，包括：

- 运行架构
- Agent 分工
- 主要输入输出结构
- 模型与向量检索方案
- 当前已实现与未实现边界

## 2. 整体架构

当前系统采用 `1 + 4` Agent 架构：

```text
Streamlit UI
    ↓
FastAPI API
    ↓
OrchestratorAgent
    ↓
CorpusAgent → RetrievalAgent → AnalystAgent → AdvisorAgent
    ↓
ChromaDB / SQLite / 外部 LLM 与嵌入服务
```

其中：

- Streamlit 负责上传、展示、复核、审计和规则配置
- FastAPI 负责参数校验和路由转发
- OrchestratorAgent 负责固定流程编排
- 4 个执行 Agent 负责单报告分析主链路

## 3. 固定工作流

### 3.1 已实现工作流

当前唯一完整打通的是：

`single_report_analysis`

执行顺序如下：

1. `CorpusAgent`：解析文件、清洗文本、分块、入库
2. `RetrievalAgent`：基于规则与 RAG 做议题识别
3. `AnalystAgent`：对照标准和同行案例做差距分析
4. `AdvisorAgent`：生成优化建议与优先行动项

### 3.2 预留工作流

以下流程在枚举中保留，但当前未实现：

- `multi_company_benchmark`
- `batch_corpus_processing`

## 4. Agent 模型

### 4.1 BaseAgent

所有 Agent 继承 `BaseAgent`，提供统一的：

- `run()`
- `_execute()`
- `reset()`

状态枚举为：

- `IDLE`
- `RUNNING`
- `SUCCESS`
- `FAILED`
- `PENDING`

说明：

- `PENDING` 在基类中存在
- 当前主流程实际主要使用 `IDLE -> RUNNING -> SUCCESS/FAILED`

### 4.2 OrchestratorAgent

职责：

- 校验 `workflow_type`
- 按固定顺序调用 4 个执行 Agent
- 聚合阶段输出
- 抛出未实现流程错误

输入：

```python
{
    "workflow_type": "single_report_analysis",
    "file_path": "..."
}
```

输出：

```python
{
    "corpus_result": {...},
    "retrieval_result": {...},
    "analyst_result": {...},
    "advisor_result": {...},
    "workflow_status": "completed"
}
```

### 4.3 CorpusAgent

职责：

- 校验上传文件
- 提取 PDF / Word / Excel 文本
- 可选调用 LLM 修复乱码或表格识别异常
- 按固定参数分块
- 写入 ChromaDB 与本地原始文本目录

输入：

```python
{
    "file_path": "C:/path/to/file.pdf"
}
```

输出核心字段：

```python
{
    "metadata": {
        "file_name": "...",
        "file_suffix": ".pdf",
        "file_size": 12345,
        "text_length": 67890,
        "chunk_count": 25,
        "processed_at": "...",
        "corpus_id": "..."
    },
    "raw_text": "...",
    "fixed_text": "...",
    "chunks": [(start, end, text)],
    "corpus_id": "..."
}
```

### 4.4 RetrievalAgent

职责：

- 热重载 `topic_rules.json`
- 从 `standards` 和 `peer_reports` 集合检索相关上下文
- 基于 Prompt + LLM 输出 `identified_topics`

输出核心字段：

```python
{
    "identified_topics": [
        {
            "topic_id": "carbon_emission",
            "topic_name": "温室气体排放",
            "evidence": "原文证据片段",
            "confidence": "high"
        }
    ],
    "retrieved_standards": [...],
    "retrieved_peers": [...],
    "coverage_summary": "...",
    "retrieval_summary": {
        "topic_count": 3,
        "standards_count": 8,
        "peers_count": 5
    }
}
```

### 4.5 AnalystAgent

职责：

- 热重载 `esg_standards.json`
- 结合议题、标准条文和同行案例生成差距分析

输出核心字段：

```python
{
    "gap_analysis": [
        {
            "topic_name": "温室气体排放",
            "current_disclosure": "...",
            "standard_requirement": "...",
            "gap": "...",
            "gap_level": "major",
            "standard_ref": "ISSB S2-29"
        }
    ],
    "peer_comparison": [
        {
            "topic_name": "温室气体排放",
            "peer_practice": "...",
            "our_position": "落后"
        }
    ],
    "overall_assessment": "...",
    "analysis_summary": {
        "major_gaps": 1,
        "minor_gaps": 2,
        "no_gaps": 0,
        "total_topics": 3
    }
}
```

### 4.6 AdvisorAgent

职责：

- 基于差距分析结果生成建议
- 输出结构化建议、优先行动项和一段完整文本

输出核心字段：

```python
{
    "recommendations": [
        {
            "topic_name": "温室气体排放",
            "action": "...",
            "priority": "high",
            "reference_case": "...",
            "standard_basis": "ISSB S2-29"
        }
    ],
    "priority_actions": ["...", "...", "..."],
    "generated_content": "...",
    "advisor_summary": {
        "total_recommendations": 4,
        "high_priority": 2,
        "medium_priority": 1,
        "low_priority": 1,
        "priority_actions_count": 3
    }
}
```

## 5. 模型与检索层

### 5.1 大语言模型

当前实现使用 OpenAI 兼容接口，通过以下配置驱动：

- `LLM_BASE_URL`
- `LLM_API_KEY`
- `LLM_MODEL`

用途：

- 文本修复
- 议题识别
- 差距分析
- 优化建议生成

### 5.2 向量嵌入模型

当前实现使用：

- 嵌入模型：`BAAI/bge-m3`
- 服务提供方：SiliconFlow

实现方式：

- Chroma 使用 `OpenAIEmbeddingFunction`
- `api_base` 指向 `https://api.siliconflow.cn/v1`

### 5.3 向量集合

当前 Chroma 中主要使用 4 个集合：

- `corpus_collection`
- `esg_metrics_collection`
- `standards`
- `peer_reports`

说明：

- `esg_metrics_collection` 的能力已预留
- 当前 `CorpusAgent` 默认不会自动抽取 ESG 指标写入该集合

## 6. 数据模型

### 6.1 文件与文本

语料解析后会落到：

- `data/raw_corpus/`：原始文本和修复文本
- `data/chroma_db/`：向量库

### 6.2 审计数据

审计日志当前写入：

- `data/sqlite_db/audit_log.db`

主要字段包括：

- `timestamp`
- `operation_type`
- `operator`
- `operation_detail`
- `status`
- `error_info`
- `log_hash`

### 6.3 检索 metadata 约束

为了兼容同行案例检索和后续对标分析，`peer_reports` 中建议至少保留：

- `company`
- `year`
- `industry`
- `topic`

## 7. 规则模型

当前规则配置目录为 `templates/rule_templates/`。

主要文件：

- `topic_rules.json`：议题定义
- `esg_standards.json`：标准配置
- `match_rules.json`：关键词/正则/语义匹配策略
- `esg_indicators.json`：指标定义
- `unit_conversions.json`：单位换算表

## 8. 前端页面模型

当前 Streamlit 页面与主流程的对应关系如下：

1. 报告上传：调用 `/corpus/process`
2. 议题识别：调用 `/retrieval/run`
3. 差距分析：调用 `/analyst/analyze`
4. 人工复核：人工确认建议并导出
5. 审计日志：查询与校验审计记录
6. 规则配置：直接修改 JSON 规则文件

说明：

- 首页数据当前是静态演示值
- 对标分析页面当前是规划说明页

## 9. 当前边界与限制

### 9.1 已实现

- 单报告分析主链路
- 审计日志查询与导出
- 规则页面可视化编辑
- 标准库与同行报告导入脚本

### 9.2 未实现

- 多企业对标工作流
- 对话式检索问答
- 标准版本差异分析

### 9.3 合规限制

- AI 输出只能作为辅助信息
- 所有对外披露内容必须人工复核
- 当前实现没有把 AI 输出视为最终结论
