# ESG 披露智能分析 MVP 实施计划

> **For agentic workers:** REQUIRED SUB-SKILL: Use `superpowers:executing-plans` to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** 交付一个本地可运行、以远景能源 2024 中文 ESG 报告为输入、按版本和生效日期管理 GRI 条款、支持证据追溯与人工复核的单报告披露分析 MVP。

**Architecture:** 保持现有 `OrchestratorAgent -> CorpusAgent -> RetrievalAgent -> AnalystAgent -> AdvisorAgent` 主链路，新增 `AnalysisRun` 作为一次分析任务的持久化业务对象。Streamlit 作为唯一 P0 操作界面；FastAPI 提供统一任务接口；标准与结论通过条款版本、页码证据、运行记录和复核记录实现可追溯。

**Tech Stack:** Python、FastAPI、Streamlit、ChromaDB、SQLite、BAAI/bge-m3、OpenAI 兼容 LLM 接口、GRI 官方 PDF。

---

## 1. 项目完成定义

### 1.1 P0 目标

面向 ESG 专员，完成一条真实可验证的单报告链路：

```text
远景 2024 中文 ESG 报告
  -> 文本解析与分块
  -> GRI 条款检索与证据定位
  -> 披露覆盖判断
  -> 差距与框架建议
  -> 人工复核
  -> 审计留痕与导出
```

### 1.2 P0 范围

- 输入：一份已核验来源的远景能源 2024 中文 ESG 报告 PDF。
- 标准：GRI 模块化标准，按 `version`、`published_at`、`effective_from` 管理。
- 当前分析：使用报告期适用条款评估已披露、部分披露、未披露、不适用和待人工复核。
- 准备度分析：将截至 2026 年已生效的新要求单列为下一期披露准备度，不倒推为 2024 年不合规。
- 观察项：将 2027 年及以后生效的条款单列为标准变更观察项。
- 界面：现有 Streamlit 页面和新增的 AnalysisRun 任务中心。

### 1.3 P0 排除项

- 不接入 `C:\Users\43480\Desktop\esg-dashboard`。
- 不将多企业实质性议题对标、实时舆情监测、登录、多用户协作、企业内部系统接入计入 P0 验收。
- 不将 AI 输出直接视为正式对外披露结论。

### 1.4 结项验收条件

- [ ] 在干净的本地环境中按文档启动 FastAPI 和 Streamlit。
- [ ] 用户可以完成一次从上传报告到导出复核结果的全流程操作。
- [ ] 每条披露判断均保留 GRI 条款 ID、版本、证据原文、报告页码、置信度和人工复核状态。
- [ ] 对固定人工标注集输出差距判断准确率、证据定位有效率、格式合规率和建议可用性结果。
- [ ] 对误判样本按数据、检索、标准映射、模型判断和人工口径分类复盘。
- [ ] README、部署说明、用户操作手册、项目模型说明和本计划的能力边界一致。

## 2. 已确认决策

| 决策项 | 已确认口径 | 影响 |
|---|---|---|
| P0 业务场景 | 单报告披露分析与下一期框架建议 | 不开发多企业与舆情主链路 |
| 企业报告 | 远景能源 2024 中文 ESG 报告 | 证据页码、人工标注与展示均以该版本为准 |
| 标准策略 | GRI 按版本和生效日期管理 | 不将未来条款误判为历史不合规 |
| 标准原文 | 英文原文为权威来源；中文用于工作译文、检索和展示 | 结论保留英文条款 ID 和中文证据 |
| P0 界面 | Streamlit | 不连接静态 React Dashboard |
| 运行对象 | `AnalysisRun` | 每次分析可以查询、复核、导出和审计 |

## 3. 生命周期阶段与阶段门

### 阶段 A：原始资料与标准范围冻结

**目的：** 建立可审计的数据起点，防止标准版本、报告版本和条款适用时间混用。

**交付物：**

- 原始报告 PDF 与来源记录。
- GRI 官方英文 PDF、SHA-256、官方 URL、下载日期和版权说明。
- `p0_gri_manifest`：条款 ID、版本、发布日期、生效日期、原文页码、中文工作译文来源、分析模式。
- 报告与标准的文件清单及哈希。

**阶段门：** 任何条款进入 Agent 前，均能回答“原文在哪里、何时生效、为何适用于本次分析”。

### 阶段 B：统一领域契约

**目的：** 将 Agent 输出、数据库记录、Streamlit 展示和导出文件统一到同一业务模型。

**核心对象：**

```text
AnalysisRun
  report_id, standard_profile_id, status, started_at, completed_at

DisclosureAssessment
  clause_id, clause_version, assessment_mode, verdict, confidence,
  evidence[], rationale, recommendation, review_status

Evidence
  corpus_id, chunk_id, source_page, source_text, relevance

ReviewDecision
  reviewer, status, edited_conclusion, note, reviewed_at
```

**阶段门：** 任意一次运行可以被完整序列化、重新展示和导出；页面字段不依赖临时 Session State 才能恢复。

### 阶段 C：GRI 知识库与双语证据层

**目的：** 将原始 PDF 转化为受版本控制的结构化条款与可追溯证据。

**交付物：**

- 结构化 GRI 条款库：英文原文、中文工作译文、版本和生效日期。
- 报告分块 metadata：`company`、`year`、`industry`、`topic`、页码、分块 ID、文件哈希。
- 术语映射表：GRI 英文术语、中文同义表达、报告常见表述。

**阶段门：** 检索结果可回到原始 PDF 页码；任何中文工作译文均标注其来源类型。

### 阶段 D：Agent 与 Prompt 约束改造

**目的：** 让现有 1+4 Agent 架构输出可验证的 GRI 条款级结果。

**规则：**

- 检索 Agent 只能返回带来源页码和分块 ID 的证据。
- 分析 Agent 的结论固定为 `disclosed`、`partial`、`missing`、`notApplicable`、`manualReview`。
- 没有足够证据时输出 `manualReview`，不得补造报告事实。
- 建议 Agent 仅基于条款要求、报告证据和已确认差距生成中文建议。
- 所有 Agent 输出在进入下一阶段前通过结构化校验。

**阶段门：** 对预设输入，输出可解析、枚举值合法、证据字段完整；无证据时不会生成确定性合规结论。

### 阶段 E：AnalysisRun 持久化、API 与 Streamlit 工作台

**目的：** 把四个 Agent 的执行过程转化为用户可操作、可恢复的分析任务。

**P0 页面行为：**

1. 创建任务并上传报告。
2. 显示解析、检索、分析、建议四阶段状态。
3. 以条款列表显示结论、证据、页码、置信度和建议。
4. 支持筛选结论、查看原文证据、标记复核状态、编辑建议。
5. 导出已复核建议文本和结构化 JSON。

**阶段门：** 刷新 Streamlit 页面后，已完成任务和复核记录仍可加载；失败任务显示失败阶段和可读原因。

### 阶段 F：真实运行、评测与误差复盘

**目的：** 将“能运行”提升为“结果可证明”。

**评测设计：**

- 从条款中按结论类别和 ESG 主题分层抽取人工标注样本。
- 标注人员记录条款、报告页码、正确结论和判定理由。
- 统计结论准确率、证据定位有效率、结构化输出合规率、人工复核通过率和建议可用性。
- 对每个误判归入标准适用性、中文术语、证据检索、模型推理、输出格式或人工口径之一。

**阶段门：** 输出可复核的评测报告；未达到 PRD 目标的指标必须附带具体误差样本与改进动作。

### 阶段 G：结项交付与求职材料转化

**目的：** 将技术实现转化为可演示、可审计、可用于简历与面试的项目包。

**交付物：**

- 更新后的 PRD、架构图、数据字典、部署说明、用户操作手册和 FAQ。
- 一次完整分析的输入、输出、审计记录、人工复核记录和评测报告。
- 3 至 5 分钟演示脚本与录屏。
- 基于真实评测指标的简历项目描述和面试问答。

**阶段门：** 第三方按文档可复现演示；所有简历指标均能追溯到评测或运行证据。

## 4. 实施顺序

1. 阶段 A：冻结资料和标准范围。
2. 阶段 B：确认 `AnalysisRun` 领域契约。
3. 阶段 C：构建 GRI 结构化条款与报告证据层。
4. 阶段 D：更新 Agent、Prompt、规则与结构化校验。
5. 阶段 E：实现任务持久化、统一 API 和 Streamlit 任务中心。
6. 阶段 F：完成真实运行、评测、误差修复和回归验证。
7. 阶段 G：同步项目文档、录制演示、更新简历口径并结项。

阶段 A 至 F 未通过阶段门时，不进入下一阶段。阶段 G 不新增功能，只整理已验证能力与证据。

## 5. 关键风险与处理原则

| 风险 | 处理原则 |
|---|---|
| 标准版本混用 | 标准 profile 必须按生效日期冻结；未来条款只能进入准备度或观察项结果 |
| 双语语义偏差 | 英文原文保留为权威依据；中文译文注明来源；低置信度进入人工复核 |
| AI 幻觉 | 证据缺失即 `manualReview`；建议不得引入证据外事实 |
| 真实结果弱于 Demo | 优先披露误差类型和改进动作，不用静态样本替换真实结果 |
| 外部模型与数据安全 | 仅使用公开报告；记录外部服务调用边界；不提交密钥或原始 `.env` |
| 范围膨胀 | 竞对、舆情和管理驾驶舱归为 P1/P2，P0 验收不依赖它们 |
