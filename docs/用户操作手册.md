# 用户操作手册

## 1. 使用前准备

### 1.1 当前可用范围

当前系统可直接使用的是 P0 fixed-report artifact-backed review workbench：

- 固定报告：远景能源 2024 中文 ESG 报告。
- 固定范围：143 条 current disclosure assessments。
- Advisor coverage：143 条。
- 当前状态：`review_status=pending`、`final_evaluation_status=pending_human_evaluation`。

该工作台支持条款级查看、人工复核保存和 CSV/JSON 导出。AI 输出仍处于人工复核前状态，不能作为正式披露结论。

### 1.2 启动系统

首次使用前，先写入 E4 pending-review 数据：

```bash
python scripts/seed_stage_e4_pending_review.py
```

启动后端：

```bash
python scripts/run_api.py
```

启动前端：

```bash
streamlit run src/ui/app.py
```

默认访问地址：

- 前端：`http://127.0.0.1:8501`
- 后端健康检查：`http://127.0.0.1:8000/health`
- API 文档：`http://127.0.0.1:8000/api/docs`

如当前 Python 环境目录无写权限，项目会优先从 `vendor/python_runtime` 加载用于启动 API 的运行依赖。该目录为项目内 runtime bridge，不进入版本库；缺失时可执行：

```bash
python -m pip install --target vendor/python_runtime "anyio>=4.0.0" "uvicorn[standard]>=0.24.0"
```

### 1.3 当前页面

当前 Streamlit 只保留 P0 交付需要的页面：

1. 首页概览
2. 报告上传
3. 条款复核
4. 审计日志

旧的议题识别、差距分析、人工复核、对标分析和规则配置页面已从 P0 交付导航中移除。后续若需要恢复完整上传后自动分析链路，应作为单独工程化任务处理。

## 2. 首页概览

首页用于说明当前 P0 交付状态和入口。

推荐操作：

1. 确认系统处于 P0 fixed-report artifact-backed review workbench 口径。
2. 从首页进入“报告上传”或“条款复核”。
3. 对外展示时说明当前数据仍为 pending human evaluation。

首页中的任何概览信息都不应理解为最终人工评测指标。

## 3. 报告上传

### 3.1 页面定位

报告上传页保留基础文件上传和语料解析入口，用于查看单文件解析能力。当前 P0 条款复核工作台不依赖用户重新上传报告后实时生成 143 条结果。

### 3.2 支持格式

- PDF
- Word
- Excel

### 3.3 操作步骤

1. 打开“报告上传”页面。
2. 选择待处理文件。
3. 点击“开始处理文件”。
4. 等待后端完成文本解析和入库。
5. 查看文件名、文件大小、文本长度、分块数量和文本预览。

限制：

- 上传页生成的是语料解析结果。
- 任意报告上传后自动稳定生成完整 143 条 GRI 核验结果尚未产品化。
- 当前正式 P0 复核以 Stage E accepted artifacts 写入的 E4 数据为准。

## 4. 条款复核

### 4.1 进入条件

进入“条款复核”页面前，应已执行：

```bash
python scripts/seed_stage_e4_pending_review.py
```

页面默认读取 E4 SQLite 中的 P0 review run。

### 4.2 页面状态

页面会显示：

- `review_status=pending`
- `final_evaluation_status=pending_human_evaluation`
- Advisor 状态：`AI-assisted recommendation pending human review`

这些状态表示当前页面使用 AI-assisted pending-review 数据。P0 产品闭环已完成固定报告 artifact-backed 的查看、复核保存和导出流程；人工评测数据另行归档，不作为当前页面完成态。

### 4.3 可执行操作

在页面中可以：

1. 选择 E4 run。
2. 查看 summary，确认 assessments=143、Advisor coverage=143。
3. 按 AI verdict、review status、GRI standard、disclosure 或关键词筛选条款。
4. 查看条款级 evidence、PDF 页码、报告页码、requirement checks 和 Advisor coverage。
5. 填写 human verdict、evidence page check、requirement gap check、advisor usefulness rating、error type、correction note 和 reviewer。
6. 保存复核记录到 SQLite。

保存复核记录只写入 `p0_review_decisions`，不会修改 AI 原始 assessment 或 Advisor 原始建议。

### 4.4 导出

页面支持下载：

- assessment CSV/JSON
- Advisor CSV/JSON
- review decisions CSV/JSON

导出文件保留 AI 原始字段、证据页码、requirement IDs、manual review reason、Advisor recommendation text 和人工复核字段。导出文件不包含阶段 F 评测指标。

### 4.5 F workbook 回流

F workbook、CSV/JSON 和指标产物当前已归档保留。需要重新导入单份复核行时，可通过脚本导入：

```bash
python scripts/import_f_human_review_results.py path/to/review_rows.json
```

导入脚本会校验 `manifest_item_id`、`assessment_id` 或 `advisor_item_id` 是否能匹配 E4 数据。未知 row key 会被拒绝并返回错误清单。

## 5. 审计日志

### 5.1 日志查询

在“审计日志”页面可以按以下条件查询：

- 操作类型
- 操作人
- 时间范围
- 返回数量

查询结果会展示：

- 时间戳
- 操作类型
- 操作人
- 状态
- 日志哈希值
- 操作详情

每条记录还支持单独做完整性校验。

### 5.2 导出

审计日志页面支持：

- 生成操作类型统计。
- 导出 JSON。
- 导出 CSV。
- 手动计算任意文本的 SHA-256 哈希值。

导出目录由系统路径配置管理，目录不存在时会按需创建。

## 6. 常见注意事项

### 6.1 AI 输出使用限制

系统生成的条款判断和 Advisor 建议都只是辅助内容，不能直接作为正式披露结论。

页面和导出不得写成：

- 最终人工验证通过
- 最终建议已确认
- 已完成最终准确率评估

### 6.2 数据存储位置

当前系统会把运行数据写入：

- `data/knowledge_base/`
- `data/runs/`
- `data/sqlite_db/`
- `data/review/`

上传解析和旧 RAG 链路可能使用：

- `data/chroma_db/`
- `data/raw_corpus/`

### 6.3 对外发送说明

使用 E4 条款复核工作台不需要重新调用外部 LLM。重新运行 Stage E/E3/E3.5 或统一 Advisor 生成脚本时，才会将报告证据片段、GRI requirement 数据和 Prompt 发送到配置的大模型服务。

### 6.4 当前未实现项

以下功能当前未作为 P0 完成能力：

- 任意报告上传后的完整 143 条自动核验。
- 多企业对标分析。
- 对话窗口或检索问答。
- 披露标准版本差异分析页面。
- 登录、多用户协作和生产级部署。
